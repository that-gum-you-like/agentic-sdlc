# Spec: framework-self-release

**Date**: 2026-05-21
**Author**: CTO-Opus (claude-opus-4-7) with Bryce
**Status**: specs
**Capability**: NEW

---

## Overview

This capability provides a release pipeline for the agentic-sdlc framework itself: `agents/release.mjs` plus `scripts/release.sh`. The pipeline bumps semver, generates a changelog from git history, commits the version bump, tags, pushes, and creates a GitHub release via `gh`. It is manual-fire only (no cron). This raises the Deployment & Release maturity dimension from 1.5 to 5.0 by giving the framework a real, repeatable release process.

---

## Requirements

### REQ-001: Semver Bump

**Statement:** The system shall provide `agents/release.mjs patch|minor|major` which bumps the version in `package.json` according to semver rules.

**Acceptance Criteria:**
- [ ] Reads current version from `package.json`
- [ ] Bumps according to the subcommand: patch (0.1.0 → 0.1.1), minor (0.1.0 → 0.2.0), major (0.1.0 → 1.0.0)
- [ ] Writes new version back to `package.json` preserving formatting (uses an in-place edit, not full re-serialize)
- [ ] Validates pre-release suffixes correctly (e.g. 1.0.0-beta.1 → 1.0.0-beta.2 on patch)
- [ ] Exits non-zero if `package.json` is missing, malformed, or has no `version` field

**Dependencies:** none

**Complexity:** S

**Value:** Critical

**Notes:** Zero-dep — manual semver parsing, not the `semver` npm package.

---

### REQ-002: Changelog Generation from git log

**Statement:** The system shall generate a CHANGELOG.md entry for the new version from `git log <last-tag>..HEAD`, grouped by conventional-commit prefix.

**Acceptance Criteria:**
- [ ] Identifies the last release tag via `git describe --tags --abbrev=0`
- [ ] If no tags exist, uses the repo's first commit as the base
- [ ] Parses commits between base and HEAD, grouping by prefix:
  - `feat:` → "### Added"
  - `fix:` → "### Fixed"
  - `docs:` → "### Documentation"
  - `refactor:`, `chore:`, `style:` → "### Internal"
  - Anything else → "### Changed"
- [ ] Prepends new entry to existing `CHANGELOG.md` (or creates it if missing)
- [ ] Entry header is `## v<version> — <ISO date>`
- [ ] Each commit becomes a bullet point: `- <description> (<short-hash>)`
- [ ] Skips merge commits

**Dependencies:** REQ-001

**Complexity:** M

**Value:** Critical

**Notes:** No external changelog tool. The format follows Keep-a-Changelog conventions but generated by simple grouping.

---

### REQ-003: Commit, Tag, Push, GitHub Release

**Statement:** The system shall commit the version + changelog changes, tag with `v<version>`, push branch + tag, and create a GitHub release via `gh`.

**Acceptance Criteria:**
- [ ] Commits `package.json` + `CHANGELOG.md` with message `chore: release v<version>`
- [ ] Tags the commit with `v<version>` (no `v` prefix doubled)
- [ ] Pushes the current branch and the new tag to origin
- [ ] Runs `gh release create v<version> --notes-file <generated-notes-file>` where notes-file is the new CHANGELOG.md section only
- [ ] If any step fails, prints actionable error and exits non-zero
- [ ] On success, prints the GitHub release URL to stdout

**Dependencies:** REQ-001, REQ-002

**Complexity:** M

**Value:** Critical

**Notes:** `gh` is already installed on Bryce's machine. The release is created as non-draft, non-prerelease by default; `--draft` flag for draft releases.

---

### REQ-004: Dry-Run Mode

**Statement:** The system shall provide a `--dry-run` flag that shows exactly what would happen without making any changes.

**Acceptance Criteria:**
- [ ] `release.mjs patch --dry-run` prints: current version → new version, generated changelog content, planned git commands, planned `gh release` command
- [ ] No files are written, no commits made, no pushes, no gh release created
- [ ] Exits 0 if all checks pass (would-succeed); non-zero otherwise (would-fail)

**Dependencies:** REQ-001, REQ-002, REQ-003

**Complexity:** S

**Value:** High

**Notes:** Dry-run is the safety net before every real release.

---

### REQ-005: Pre-Flight Checks

**Statement:** The system shall refuse to proceed with a release unless the working tree is clean, the current branch is `main`, all tests pass, and `gh` is authenticated.

**Acceptance Criteria:**
- [ ] Checks `git status --porcelain` is empty (no staged or unstaged changes)
- [ ] Checks `git rev-parse --abbrev-ref HEAD` returns `main`
- [ ] Runs `node tests/adapter-and-model-manager.test.mjs` (or the configured test command), exits non-zero on failure
- [ ] Runs `gh auth status`, exits non-zero if not authenticated
- [ ] Any failed check prints what was wrong and exits before any mutation
- [ ] Flag `--skip-tests` bypasses the test check (with a warning); other checks are mandatory

**Dependencies:** REQ-001, REQ-003

**Complexity:** S

**Value:** Critical

**Notes:** "Wrote the code" is not "shipped" — your memory rule about the Done Checklist applies. The release pipeline IS the framework's done-checklist enforcement.

## Acceptance Criteria (Scenarios)

### Scenario 1: Patch Release Happy Path

**Verifies:** REQ-001, REQ-002, REQ-003, REQ-005

**WHEN** Bryce runs `node agents/release.mjs patch` on a clean main branch with passing tests, current version 0.1.0, and 5 conventional commits since last tag

**THEN** Version becomes 0.1.1 in `package.json`

**AND** `CHANGELOG.md` has a new `## v0.1.1 — <today>` section listing 5 commits grouped by prefix

**AND** A commit `chore: release v0.1.1` is created with both files

**AND** Tag `v0.1.1` is created and pushed

**AND** A GitHub release at `v0.1.1` exists with the changelog body, URL printed to stdout

---

### Scenario 2: Dry Run Shows Plan Without Changes

**Verifies:** REQ-004

**WHEN** Bryce runs `node agents/release.mjs minor --dry-run`

**THEN** stdout shows: current version, new version (minor bump), generated changelog content, planned git/gh commands

**AND** `package.json` and `CHANGELOG.md` are unchanged

**AND** `git status` is unchanged

**AND** No tag is created

---

### Scenario 3: Pre-Flight Failure Blocks Release

**Verifies:** REQ-005

**WHEN** Bryce runs `node agents/release.mjs patch` on a branch with uncommitted changes

**THEN** Output includes "working tree is not clean: commit or stash first"

**AND** Exit code is non-zero

**AND** `package.json` is unchanged

---

### Scenario 4: Error Case — Wrong Branch

**Verifies:** REQ-005

**WHEN** Bryce runs `node agents/release.mjs patch` on a feature branch

**THEN** Output includes "release must be cut from main; current branch is <name>"

**AND** Exit code is non-zero

---

### Scenario 5: Edge Case — No Previous Tag

**Verifies:** REQ-002

**WHEN** Bryce runs `release.mjs patch` on a repo with zero existing tags

**THEN** The changelog includes all commits from the first commit to HEAD

**AND** A tag `v<version>` is created

**AND** Subsequent releases use this tag as the base

---

## Invariants

- The pipeline never mutates files on failure — either the full release succeeds or no changes are committed
- A release commit always contains exactly two files: `package.json` and `CHANGELOG.md` (no other staged changes)
- The tag exactly matches the version with a `v` prefix: tag `v0.2.3` ↔ version `0.2.3`
- GitHub release notes match the CHANGELOG.md section for that version byte-for-byte (excluding the version header)

---

## Out of Scope

- npm publish (the framework is zero-dep and not consumed via npm)
- Automatic release scheduling (manual-fire only by decision D7)
- Multi-package monorepo releases
- Pre-release tracks (beta/rc channels) — patch/minor/major only
